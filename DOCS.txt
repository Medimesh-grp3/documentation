## 1. Executive Summary

MediMesh is a cloud-native hospital management platform built on a microservices architecture using the MERN stack (MongoDB, Express.js, React, Node.js). The system is designed to digitize and streamline core hospital operations including patient registration, doctor management, appointment scheduling, vitals tracking, pharmacy inventory, ambulance fleet management, complaint resolution, and community forums.

The platform comprises eleven distinct services -- nine domain microservices, one Backend-for-Frontend (BFF) aggregation layer, and one React single-page application frontend -- each independently deployable and horizontally scalable. The entire system is containerized with Docker, orchestrated on Kubernetes (deployed via kubeadm on AWS EC2), and delivered through a fully automated CI/CD pipeline using GitHub Actions with GitOps-based continuous deployment via ArgoCD.

The infrastructure leverages modern Kubernetes patterns including the Gateway API (kGateway with Envoy Proxy) for traffic routing, NFS-backed dynamic persistent storage for MongoDB StatefulSets, Horizontal Pod Autoscaling, and network policies for zero-trust security. An external HAProxy load balancer provides the public entry point, distributing traffic across the Kubernetes worker nodes.

This document provides a comprehensive technical and business-level overview of the entire system, its architecture, deployment workflows, operational procedures, and governance framework.

## 2. Business Requirements and Problem Statement

### 2.1 Problem Statement

Traditional hospital management relies on fragmented, paper-based or monolithic software systems that suffer from several critical limitations. These systems cannot scale independently to meet varying demand across departments, they present single points of failure, and they make it difficult to adopt modern deployment practices such as continuous delivery and zero-downtime updates. Hospitals require a platform that can manage diverse operational domains -- from patient intake to pharmacy inventory -- while maintaining high availability, data consistency, and operational agility.

### 2.2 Business Objectives

The MediMesh platform addresses these challenges by delivering a system that separates concerns into independently deployable microservices, each owning its own data store. The key business objectives are as follows.

**Operational Efficiency.** Automate appointment workflows (booking, approval, rejection), vitals recording, pharmacy stock management, and ambulance dispatch tracking to reduce manual overhead and human error.

**Role-Based Access.** Enforce clear separation of responsibilities across three user roles -- administrators, doctors, and patients -- ensuring that sensitive operations such as pharmacy inventory management and complaint resolution are restricted to authorized personnel.

**High Availability.** Ensure the platform remains operational through redundant service replicas, a three-node MongoDB ReplicaSet with automatic failover, and externalized load balancing. The system targets near-zero downtime during deployments through rolling updates.

**Scalability.** Allow individual services to scale independently based on demand. The frontend, which faces the highest user concurrency, supports automatic horizontal scaling from two to five pods based on CPU utilization.

**Rapid and Safe Delivery.** Enable engineering teams to ship changes to any individual service without affecting the rest of the platform, supported by a multi-stage CI/CD pipeline with security scanning, manual approval gates, and automated GitOps deployment.

## 3. Architecture Overview (High-Level Design)

### 3.1 System Architecture

MediMesh follows a microservices architecture pattern with a Backend-for-Frontend aggregation layer. The system is composed of the following tiers.

**Presentation Tier.** A React single-page application built with Create React App, served by Nginx in production. The frontend communicates exclusively with the BFF layer via a reverse proxy configured in Nginx.

**Aggregation Tier.** The BFF (Backend-for-Frontend) service acts as a central API layer for the frontend. It receives all API requests from the user interface, verifies that the user is authenticated, and then forwards each request to the correct backend microservice. For dashboard views, it collects data from multiple services at once and returns a combined response.

**Service Tier.** Nine domain microservices, each built on Express.js and Mongoose. Services communicate synchronously via HTTP/REST through internal Kubernetes DNS.

**Data Tier.** A three-node MongoDB ReplicaSet deployed as a Kubernetes StatefulSet with NFS-backed persistent volumes. The ReplicaSet provides automatic failover (one PRIMARY, two SECONDARY nodes) and data replication via the oplog.

### 3.2 Traffic Flow

```
Internet (Port 80)
       |
  HAProxy (Separate EC2 Instance)
  Round-robin load balancing
       |
  kGateway (Envoy Proxy)
  Gateway API HTTPRoute -- path-based routing
       |
  +-----------+-----------+-----------+
  |           |           |           |
Frontend   BFF       Auth        [Other Services]
(React)   (5010)    (5001)       (5002-5009)
  |           |           |           |
  |      +---------+  +---------+  +---------+
  |      | Service | ->| MongoDB ReplicaSet  |
  |      | Calls   |  | 3-node (rs0)         |
  |      +---------+  +---------------------+
```

The frontend catch-all route (`/`) is placed last in the HTTPRoute rules, ensuring that specific API path prefixes (`/api`, `/auth`, `/doctor`, etc.) match first and route to their respective backend services.

## 4. Detailed Design (Low-Level Design)

### 4.1 Authentication Service (medimesh-auth)

The authentication service is the identity backbone of the platform. It manages user registration, credential validation, and JWT token issuance.

**Data Model.** The User schema stores `username` (unique), `password` (bcrypt-hashed with cost factor 10), `role` (enum: admin, doctor, patient), `fullName`, `email`, and `phone`. Timestamps are automatically tracked via Mongoose.

**Password Security.** A Mongoose pre-save hook intercepts password modifications and applies bcrypt hashing before persistence. Password comparison during login uses `bcrypt.compare` against the stored hash.

**Token Issuance.** Upon successful authentication, the service generates a JWT containing `userId`, `username`, `role`, and `fullName` with a 24-hour expiration. The signing secret is injected via the `JWT_SECRET` environment variable, sourced from a Kubernetes Secret in production.

**Admin Protection.** Registration of admin-role users is explicitly blocked at the API level. The only admin account is seeded at service startup (username: `admin`, password: `123456`). Five default doctor accounts (`doctor1` through `doctor5`, password: `pass123`) are also seeded.

**Token Validation.** A dedicated `/api/auth/validate` endpoint allows other services to verify token validity without direct database access.

### 4.2 Backend-for-Frontend (medimesh-bff)

The BFF service serves as the centralized entry point for all frontend API calls. It performs two key functions.

**Request Forwarding.** When the frontend makes an API call, the BFF receives it, attaches the user's authentication credentials, and forwards it to the appropriate backend microservice. It maintains a registry of all backend service URLs configured via environment variables.

**Dashboard Aggregation.** The `/api/dashboard` endpoint collects data from multiple backend services in parallel and returns a combined response tailored to the user's role. For administrators, it gathers doctors, appointments, medicines, ambulances, and complaints. For doctors, it gathers their appointments and vitals. For patients, it gathers their appointments, available doctors, ambulances, and pharmacy inventory. If any individual service is temporarily unavailable, the dashboard still displays data from the remaining services.

### 4.3 Domain Microservices Pattern

All nine domain microservices follow an identical structural pattern consisting of a server entry point (`server.js`), Mongoose models (`models/`), Express route handlers (`routes/`), and JWT middleware (`middleware/auth.js`).

Each service implements a health check endpoint at `/health` returning `{ status: 'ok', service: '<service-name>' }`, used by Kubernetes readiness and liveness probes.

**Retry Logic.** Every service implements exponential backoff for MongoDB connections with up to five retry attempts. The initial delay is three seconds, doubling with each subsequent attempt (3s, 6s, 12s, 24s, 48s). If all retries are exhausted, the process exits with code 1, allowing Kubernetes to restart the pod.

### 4.4 Frontend Architecture

The frontend is a React 18 single-page application using React Router v6 for client-side routing and Axios for HTTP communication. It is built with Create React App and served by Nginx in production.

The Nginx configuration implements two key routing behaviors. Static asset requests are served from `/usr/share/nginx/html` with a fallback to `index.html` for client-side routing support. API requests matching `/api/` are reverse-proxied to the BFF service at `http://medimesh-bff-svc:5010`, leveraging Kubernetes internal DNS for service discovery.

The user interface follows a clean, professional blue-themed design with clearly distinguishable status indicators for appointment and complaint workflows (approved, pending, and rejected states).

## 5. Infrastructure Setup

### 5.1 Cloud Provider and Topology

The platform is deployed on Amazon Web Services (AWS) using EC2 instances within a single Virtual Private Cloud (VPC). The infrastructure consists of the following components.

| Component | Instance Type | Role |
|-----------|--------------|------|
| Kubernetes Master | EC2 (Ubuntu 22.04+) | Control plane (kubeadm) |
| Kubernetes Worker 1 | EC2 (Ubuntu 22.04+) | Workload execution |
| Kubernetes Worker 2 | EC2 (Ubuntu 22.04+) | Workload execution |
| HAProxy and NFS Server | EC2 (t2.micro/t3.micro) | External load balancer and NFS storage |

All instances reside in the same VPC to enable private network communication. The Kubernetes cluster is bootstrapped using kubeadm, providing a production-grade control plane without the abstraction of managed Kubernetes services.

### 5.2 NFS Storage Infrastructure

Persistent storage for MongoDB is provided by an NFS server running on the HAProxy EC2 instance. The NFS export path is `/srv/nfs/medimesh`, accessible to all nodes in the VPC CIDR range (e.g., `172.31.0.0/16`).

The Kubernetes-side storage stack consists of three components. A `ServiceAccount` with RBAC permissions (`nfs-client-provisioner`) runs the provisioner pod. A `StorageClass` named `nfs-dynamic` (set as the default) configures the provisioner with `onDelete: retain` to archive data rather than delete it, and uses a `pathPattern` of `${.PVC.namespace}-${.PVC.name}` for organized subdirectory naming. The provisioner deployment (`nfs-subdir-external-provisioner v4.0.2`) watches for PVC creation events and dynamically provisions NFS-backed PersistentVolumes.

### 5.3 Security Groups

## 6. Kubernetes Implementation (Deep Dive)

### 6.1 Namespace Isolation

The platform uses namespace isolation to separate concerns. In the raw Kubernetes manifests, a single `medimesh` namespace houses all resources. The Helm-based deployment introduces a more mature three-namespace model.

| Namespace | Purpose | Resources |
|-----------|---------|-----------|
| medimesh-frontend | Presentation tier | Frontend Deployment, ClusterIP Service |
| medimesh-backend | Application tier | BFF + 9 microservice Deployments, ClusterIP Services |
| medimesh-db | Data tier | MongoDB StatefulSet, NFS Provisioner |

### 6.2 Workload Definitions

**Backend Deployments.** Each microservice runs as a Kubernetes Deployment with two replicas. The deployment strategy is RollingUpdate with `maxUnavailable: 1` and `maxSurge: 1`, ensuring at least one pod remains available during updates. All backend deployments include an `initContainer` (`busybox:1.36`) that performs a TCP connectivity check against `medimesh-mongodb-0.medimesh-mongodb:27017`, blocking the main container startup until MongoDB is reachable.

**Resource Allocation.** Every container specifies resource requests and limits.

| Container Type | CPU Request | CPU Limit | Memory Request | Memory Limit |
|---------------|------------|----------|---------------|-------------|
| Backend Service | 100m | 500m | 128Mi | 512Mi |
| MongoDB | 250m | 500m | 256Mi | 512Mi |
| Frontend | 100m | 500m | 128Mi | 512Mi |
| Log Sidecar | 25m | 50m | 32Mi | 64Mi |

**Health Probes.** All services expose HTTP-based readiness and liveness probes. Backend services probe `/health` on their respective ports. The frontend probes `/` on port 80. MongoDB uses `exec`-based probes: the readiness probe executes a `mongosh` command to verify ReplicaSet membership state (PRIMARY or SECONDARY), while the liveness probe executes `db.adminCommand('ping')`.

### 6.3 MongoDB StatefulSet

MongoDB is deployed as a three-replica StatefulSet with the following characteristics.

**Stable Network Identity.** A headless Service (`clusterIP: None`) provides stable DNS names for each pod: `medimesh-mongodb-0.medimesh-mongodb`, `medimesh-mongodb-1.medimesh-mongodb`, and `medimesh-mongodb-2.medimesh-mongodb`.

**ReplicaSet Configuration.** Each MongoDB pod is started with `--replSet rs0 --bind_ip_all`. After all three pods are running, a Kubernetes Job (`mongodb-rs-init`) executes `rs.initiate()` with `medimesh-mongodb-0` configured with priority 2 (preferred PRIMARY) and the other two with priority 1.

**Pod Anti-Affinity.** Preferred pod anti-affinity ensures MongoDB pods are distributed across different worker nodes for fault tolerance. Tolerations allow scheduling on the control plane node if worker capacity is insufficient.

**Persistent Storage.** Each pod receives a 5Gi PVC via `volumeClaimTemplates`, dynamically provisioned by the NFS StorageClass. The `ReadWriteOnce` access mode ensures data integrity.

**Replication Verification.** The init Job includes a replication test that writes a document to the PRIMARY, waits three seconds, and reads it from both SECONDARYs to confirm oplog replication is functioning.

### 6.4 Gateway API (kGateway)

The platform uses the Kubernetes Gateway API standard (`gateway.networking.k8s.io/v1`) with kGateway (Envoy Proxy-based) as the implementation.

**Gateway Resource.** A single Gateway named `medimesh-gateway` listens on port 80 (HTTP) with the `kgateway` GatewayClass. It restricts route attachment to the `Same` namespace.

**HTTPRoute Rules.** An HTTPRoute named `medimesh-routes` defines eleven path-based routing rules using `PathPrefix` matching. The rules are ordered so that specific prefixes (`/api`, `/auth`, `/user`, `/doctor`, `/appointment`, `/vitals`, `/pharmacy`, `/ambulance`, `/complaint`, `/forum`) are evaluated before the catch-all `/` prefix that routes to the frontend. Each rule maps to a specific ClusterIP service and port.

The kGateway controller automatically creates a NodePort Service for external access. The NodePort value is dynamically assigned and must be noted for HAProxy configuration.

### 6.5 Horizontal Pod Autoscaler

The frontend Deployment is the target of an HPA (`autoscaling/v2`) that scales between 2 and 5 replicas based on CPU utilization. The target average utilization is 60 percent. This requires the Kubernetes Metrics Server to be installed and operational.

### 6.6 Frontend Sidecar Pattern

The frontend pod runs two containers side by side. The primary container serves the React application through Nginx. A lightweight sidecar container runs alongside it to continuously stream the Nginx access and error logs, making them available for centralized log collection and troubleshooting. Both containers share a temporary volume for log file access.

## 7. CI/CD Pipeline

### 7.1 Pipeline Architecture

The CI/CD system is built on GitHub Actions with a reusable workflow template architecture. A centralized repository (`Medimesh-grp3/template-ci`) contains seven reusable workflow templates that individual service repositories invoke via `workflow_call`.

Each service repository (e.g., `medimesh-auth`, `medimesh-frontend`) contains a single workflow file that orchestrates the pipeline stages by chaining calls to these shared templates. This design eliminates duplication and ensures consistent pipeline behavior across all eleven services.

### 7.2 Pipeline Stages

The pipeline executes the following stages in sequence.

**Stage 1: SonarQube Code Quality Analysis.** The `template-sonarqube.yml` workflow performs static code analysis using the SonarSource scan action. Each service has a unique SonarQube project key. Failure at this stage blocks all subsequent stages and triggers an email alert.

**Stage 2: Build and Snyk Security Scan.** The `template-build-snyk.yml` workflow installs NPM dependencies (`npm install --no-audit --no-fund`), executes the build script if present, and then runs a Snyk vulnerability scan against the dependency tree with `--severity-threshold=high`. The Snyk scan is configured with `continue-on-error: true` to prevent non-critical vulnerability findings from blocking the pipeline. Scan results are converted to HTML and uploaded as GitHub Actions artifacts with a 14-day retention period.

**Stage 3: Docker Build, Trivy Scan, and Push.** The `template-docker.yml` workflow determines a semantic version using the `mathieudutour/github-tag-action` with service-specific tag prefixes (e.g., `medimesh-auth/v1.2.3`). It supports automatic version bumping based on commit message conventions: `feature:` prefixes trigger minor bumps, `breaking:` prefixes trigger major bumps, and all other commits trigger patch bumps. The Docker image is built with three tags: semantic version (`vX.Y.Z`), version plus short SHA (`vX.Y.Z-abc1234`), and `latest`. After building, Trivy scans the image for CRITICAL and HIGH severity vulnerabilities (with `continue-on-error: true`). On `push` events to main (not pull requests), all three tags are pushed to Docker Hub.

**Stage 4: Approval Gate.** The `template-approval-gate.yml` workflow pauses the pipeline and requires human approval through a GitHub Environment named `Prod`. An email notification is sent to the configured alert address with deployment details and a direct link to the approval page. The approver can either approve (proceeding to deployment) or reject (stopping the pipeline).

**Stage 5: Continuous Deployment.** The `template-cd.yml` workflow checks out the manifest repository (`Medimesh-grp3/manifest_repo`), installs `yq`, and updates the `image.tag` field in the service-specific `values.yaml` file. It commits and pushes the change with a descriptive message (e.g., `cd(medimesh-auth): image -> v1.2.3`). The push includes retry logic with exponential backoff (five attempts with `git pull --rebase` between retries) to handle concurrent manifest updates from other service pipelines.

**Stage 6: Email Notifications.** Success and failure email notifications are sent at multiple stages using the `dawidd6/action-send-mail` action via Gmail SMTP. The final success email provides a comprehensive summary of all pipeline stages and the deployed image tags.

### 7.3 Pipeline Secrets

| Secret | Purpose |
|--------|---------|
| DOCKERHUB_USERNAME / DOCKERHUB_TOKEN | Docker Hub authentication |
| SNYK_TOKEN | Snyk security scanning |
| SONARQUBE_TOKEN / SONARQUBE_URL | SonarQube code quality |
| SMTP_USERNAME / SMTP_PASSWORD | Email notifications |
| ALERT_EMAIL | Notification recipient |
| MANIFEST_REPO_PAT | GitHub PAT for manifest repo commits |

## 8. GitOps Deployment (ArgoCD)

### 8.1 GitOps Model

MediMesh implements a GitOps deployment model where the desired cluster state is declared in a Git repository (`Medimesh-grp3/ArgoCDrepo`), and ArgoCD continuously reconciles the live cluster state to match.

The manifest repository is structured with per-service Helm value files under `helm/medimesh/charts/<service>/values.yaml`. Each file specifies the service's image repository, tag, replica count, resource limits, and service-specific configuration. When the CI/CD pipeline updates the `image.tag` field in a service's `values.yaml`, ArgoCD detects the change and automatically triggers a deployment within approximately three minutes (default sync interval).

### 8.2 Repository Structure

The manifest repository mirrors the service architecture.

```
manifest_repo/
  helm/
    medimesh/
      charts/
        auth/         -- values.yaml for medimesh-auth
        user/         -- values.yaml for medimesh-user
        doctor/       -- values.yaml for medimesh-doctor
        appointment/  -- values.yaml for medimesh-appointment
        vitals/       -- values.yaml for medimesh-vitals
        pharmacy/     -- values.yaml for medimesh-pharmacy
        ambulance/    -- values.yaml for medimesh-ambulance
        complaint/    -- values.yaml for medimesh-complaint
        forum/        -- values.yaml for medimesh-forum
        bff/          -- values.yaml for medimesh-bff
        frontend/     -- values.yaml for medimesh-frontend
        mongo/        -- values.yaml for MongoDB
        nfs/          -- values.yaml for NFS provisioner
```

### 8.3 ArgoCD Application Configuration

A parallel ArgoCD repository (`ArgoCDrepo`) contains the Helm chart that ArgoCD uses to manage the cluster. This chart references the manifest repository's values files and the subchart templates from the umbrella Helm chart, providing ArgoCD with the complete desired state for all services.

### 8.4 Sync Workflow

1. Developer pushes code to a service repository (e.g., `medimesh-auth`)
2. GitHub Actions CI pipeline runs: SonarQube, Build, Snyk, Docker build/push, Trivy
3. Human approves deployment via GitHub Environment approval (manual)
4. CD stage updates `helm/medimesh/charts/auth/values.yaml` in `ArgoCDrepo` with new image tag
5. ArgoCD detects the Git change within its polling interval
6. ArgoCD syncs the cluster state, performing a rolling update of affected pods
7. Kubernetes executes the rolling update strategy (maxUnavailable: 1, maxSurge: 1)

## 9. Deployment Strategies

### 9.1 Rolling Updates (Primary Strategy)

The primary deployment strategy across all services is RollingUpdate, configured with `maxUnavailable: 1` and `maxSurge: 1`. This means Kubernetes will create one new pod with the updated image before terminating one old pod, ensuring at least one pod remains available throughout the deployment process.

Readiness probes prevent traffic from being routed to new pods until they pass health checks (HTTP GET to `/health`), providing zero-downtime deployments under normal conditions.

### 9.2 Blue-Green with Argo Rollouts (Advanced Strategy)

The template repository includes a `template-oidc-deploy.yml` workflow that supports blue-green deployment via Argo Rollouts. This advanced strategy uses OIDC-based authentication from GitHub Actions to directly interact with the Kubernetes cluster.

In this model, a new version deploys as a "preview" (green) ReplicaSet while the active (blue) ReplicaSet continues serving traffic. After manual verification, an operator promotes the green version to active using `kubectl argo rollouts promote`. This strategy is available as an alternative deployment path and requires the Argo Rollouts controller to be installed on the cluster.

## 10. Security Architecture

### 10.1 Authentication and Authorization

**JWT-Based Authentication.** All API endpoints (except login and registration) require a valid JWT in the `Authorization: Bearer <token>` header. The BFF service validates tokens before forwarding requests to downstream services.

**Password Hashing.** User passwords are hashed using bcrypt with a cost factor of 10 before storage. Plaintext passwords are never persisted or transmitted beyond the initial registration and login requests.

**Role-Based Access Control.** Three roles (admin, doctor, patient) enforce authorization at the API route level. The `authorizeRoles()` middleware checks `req.user.role` against allowed roles, returning HTTP 403 for unauthorized access.

**Admin Registration Block.** The registration endpoint explicitly rejects any attempt to create admin-role accounts, ensuring the admin account can only be provisioned through the database seed process.

### 10.2 Kubernetes Security

**Secrets Management.** Sensitive values (JWT secret, admin credentials, MongoDB credentials) are stored in Kubernetes Secrets as base64-encoded data. Secrets are referenced in pod specifications via `secretKeyRef`, injected as environment variables at runtime.

**RBAC.** The NFS provisioner operates under a dedicated ServiceAccount (`nfs-client-provisioner`) with precisely scoped ClusterRole permissions (limited to PV/PVC/StorageClass/Event operations) and namespace-scoped Role for leader election endpoint management.

**Network Policies (Helm Deployment).** The Helm chart defines a comprehensive zero-trust network policy model. Default deny policies block all ingress and egress in all three namespaces. Explicit allow policies are then applied: frontend can reach BFF only, BFF can reach backend services, backend services can reach MongoDB on port 27017 only, MongoDB pods can communicate with each other for replication, DNS egress (UDP/TCP port 53) is allowed for all namespaces, and NFS provisioner egress is restricted to the NFS server IP (ports 2049 and 111) and the Kubernetes API server (ports 443 and 6443).

### 10.3 Container Security

**Minimal Base Images.** Backend services use `node:20-alpine`, the smallest Node.js image variant. The frontend uses a multi-stage build: `node:20-alpine` for building, `nginx:alpine` for serving.

**Package Manager Removal.** Backend Dockerfiles remove npm, yarn, and corepack after dependency installation (`RUN rm -rf /usr/local/lib/node_modules/npm ...`), reducing the attack surface.

**Security Scanning.** The CI pipeline performs two levels of security scanning: Snyk scans application dependencies for known vulnerabilities, and Trivy scans the final container image for OS and application-level vulnerabilities at CRITICAL and HIGH severity.

**System Updates.** Dockerfiles execute `apt update && apt upgrade` to apply the latest security patches to the base image.

## 11. Observability

### 11.1 Current State

The platform implements observability through the following mechanisms.

**Health Endpoints.** Every service exposes a `/health` endpoint that returns JSON status information. These endpoints serve dual purposes: Kubernetes probe targets and manual health verification.

**Kubernetes Probes.** Readiness probes (checking `/health` every 10 seconds with a 5-second timeout) ensure that pods only receive traffic when fully initialized. Liveness probes (checking `/health` every 20 seconds) detect hung or deadlocked processes and trigger automatic restarts.

**Container Logging.** Application logs are written to stdout/stderr and collected by the Kubernetes container runtime. The frontend sidecar pattern streams Nginx access and error logs to stdout, making them available through `kubectl logs`.

**HAProxy Statistics.** The HAProxy stats dashboard at `http://<HAPROXY_IP>:8404/stats` provides real-time visibility into backend health, connection counts, response times, and error rates. Access is protected by HTTP basic authentication (admin / medimesh2026).

### 11.2 Metrics and Monitoring (Prometheus and Grafana)

**Prometheus** is deployed on the cluster to collect application-level and infrastructure metrics. It scrapes health and performance data from all microservices, Node Exporter provides host-level metrics (CPU, memory, disk, network), and kube-state-metrics exposes the state of Kubernetes objects (pod status, deployment replicas, node conditions). Prometheus serves as the central metrics store and alerting engine for the platform.

**Grafana** is deployed alongside Prometheus to provide visual dashboards for operational monitoring. Pre-built dashboards display MongoDB replica set health, per-service latency distributions, cluster resource utilization, and pod status across namespaces. Grafana connects to Prometheus as its primary data source, providing real-time and historical visibility into platform health.

### 11.3 Recommended Enhancements (Not Yet Implemented)

**Distributed Tracing.** Integrate OpenTelemetry or Jaeger to trace requests across the BFF-to-microservice call chain, enabling root cause analysis for latency issues.

**Centralized Logging.** Deploy a Loki stack for log aggregation, search, and alerting across all services.

## 12. Performance and Scalability

### 12.1 Horizontal Scaling

The frontend supports automatic horizontal scaling via the HPA, scaling from 2 to 5 pods when average CPU utilization exceeds 60 percent. Backend services are configured with 2 static replicas. The MongoDB StatefulSet runs 3 replicas for data redundancy.

**Assumption:** Backend services can be configured with HPAs by adding similar `autoscaling/v2` resources targeting their respective Deployments. This is not currently implemented but would follow the same pattern as the frontend HPA.


### 12.2 Database Scalability

The three-node MongoDB ReplicaSet supports read scaling through secondary reads. Application services connect via the full replica set connection string (`mongodb://host-0:27017,host-1:27017,host-2:27017/<db>?replicaSet=rs0`), enabling automatic failover to a new PRIMARY if the current one becomes unavailable.

### 12.3 Total Pod Capacity

In the default configuration, the cluster runs approximately 24 pods: 3 MongoDB, 20 backend/BFF/frontend replicas (10 services x 2 replicas each), and 1 Envoy proxy (auto-created by kGateway). With frontend HPA scaling, this can reach up to 27 pods.

## 13. Networking Architecture

### 13.1 External Traffic Path

External traffic enters through HAProxy on port 80. HAProxy uses a round-robin algorithm to distribute requests across both Kubernetes worker nodes at the kGateway NodePort. Health checks (`GET /` with HTTP 1.1) run every 5 seconds with a fall count of 3 and rise count of 2.

### 13.2 Internal Service Discovery

Services communicate using Kubernetes DNS. In the single-namespace deployment, services are addressed as `http://medimesh-<service>-svc:<port>`. In the Helm multi-namespace deployment, fully qualified domain names are used: `http://medimesh-<service>-svc.medimesh-backend.svc.cluster.local:<port>`.

### 13.3 Service Types

| Service | Type | Purpose |
|---------|------|---------|
| All backend + frontend | ClusterIP | Internal communication only |
| MongoDB headless | ClusterIP: None | StatefulSet pod DNS identity |
| kGateway | NodePort (auto-created) | External access from HAProxy |

### 13.4 Network Policy Model (Helm)

The Helm-based deployment enforces a zero-trust network model with default-deny policies and explicit allow rules. The traffic flow chain is: External -> Gateway -> Frontend -> BFF -> Backend Services -> MongoDB. Cross-tier traffic that does not follow this chain is denied.

## 14. Configuration Management

### 14.1 ConfigMaps

The `medimesh-config` ConfigMap stores non-sensitive configuration values including `NODE_ENV` (production), MongoDB host and port, and all internal service URLs. These values are injected into pods either individually via `configMapKeyRef` or in bulk via `envFrom: configMapRef`.

### 14.2 Secrets

The `medimesh-secrets` Secret stores base64-encoded sensitive values: JWT signing secret, admin credentials, and MongoDB root credentials. These are injected into pods via `secretKeyRef` and are never stored in application code or container images.

### 14.3 Helm Values (Single Source of Truth)

The Helm umbrella chart's `values.yaml` serves as the single source of truth for all configurable parameters. It defines global settings (image registry, default tag, NFS configuration, MongoDB connection) and per-service overrides (replica count, image tag, resource limits, additional environment variables). This centralization eliminates the need to manage separate YAML manifests for each service.

## 15. Runbook (Operations Guide)

### 15.1 Initial Cluster Deployment

1. Provision EC2 instances as described in Section 5.1
2. Bootstrap Kubernetes cluster with kubeadm on master and join workers
3. Install NFS server on HAProxy instance: `sudo ./nfs-server-setup.sh 172.31.0.0/16`
4. Install NFS client on all K8s nodes: `sudo apt-get install -y nfs-common`
5. Apply namespace: `kubectl apply -f k8s/namespace.yaml`
6. Apply ConfigMap and Secrets: `kubectl apply -f k8s/configmap.yaml -f k8s/secret.yaml`
7. Deploy NFS provisioner: `kubectl apply -f k8s/nfs/rbac.yaml -f k8s/nfs/storageclass.yaml -f k8s/nfs/nfs-provisioner-deployment.yaml`
8. Deploy MongoDB StatefulSet: `kubectl apply -f k8s/mongodb/mongodb-statefulset.yaml`
9. Wait for 3 MongoDB pods, then initialize ReplicaSet: `kubectl apply -f k8s/mongodb/mongodb-rs-init-job.yaml`
10. Deploy backend services and ClusterIP services: `kubectl apply -f k8s/backend-services/ -f k8s/services/cluster-ip-services.yaml`
11. Deploy frontend: `kubectl apply -f k8s/frontend/frontend-deployment.yaml`
12. Install kGateway via Helm and apply Gateway/HTTPRoute: `kubectl apply -f k8s/gateway/kgateway.yaml`
13. Configure HAProxy with the kGateway NodePort: `sudo ./haproxy-setup.sh <NODEPORT>`

### 15.2 Common Operations

**Check all pod status:** `kubectl get pods -n medimesh -o wide`

**View service logs:** `kubectl logs -n medimesh deployment/medimesh-auth --tail=100`

**Force restart a service:** `kubectl rollout restart deployment/medimesh-auth -n medimesh`

**Scale a service:** `kubectl scale deployment/medimesh-bff --replicas=3 -n medimesh`

**Verify MongoDB ReplicaSet:**
```
kubectl exec -n medimesh medimesh-mongodb-0 -- mongosh --quiet --eval \
  "rs.status().members.forEach(m => print(m.name + ' -> ' + m.stateStr))"
```



### 15.3 Troubleshooting

**Pod stuck in CrashLoopBackOff.** Check logs with `kubectl logs <pod> -n medimesh --previous`. Common cause: MongoDB not reachable. Verify the initContainer logs and MongoDB pod status.

**MongoDB PRIMARY not elected.** Check the init Job logs: `kubectl logs -n medimesh job/mongodb-rs-init`. Re-run the Job if needed: delete and re-apply the Job manifest.

**HAProxy returns 503.** Verify kGateway pod is running and the NodePort is correct. Check HAProxy stats dashboard for backend health status.

## 16. Challenges and Solutions

| Challenge | Solution Implemented |
|-----------|---------------------|
| Service ordering dependency (services start before MongoDB) | InitContainers with TCP connectivity check; exponential backoff retry logic in application code |
| Persistent storage across worker nodes | NFS-backed dynamic provisioning with external NFS server on the HAProxy instance |
| MongoDB ReplicaSet initialization timing | Dedicated Kubernetes Job with idempotent logic (checks existing RS state before initiating) |
| Concurrent CI/CD manifest updates | Git push retry with exponential backoff and `git pull --rebase` in the CD template |
| External access without cloud load balancer | HAProxy on separate EC2 instance with round-robin to worker NodePorts |
| Cross-service authentication consistency | Shared JWT secret via Kubernetes Secret, identical middleware across all services |
| Pipeline duplication across services | Reusable GitHub Actions workflow templates in a centralized repository |
| Frontend log visibility | Sidecar container pattern tailing Nginx logs to stdout |

## 17. Future Enhancements

**Service Mesh Integration.** Adopt Istio or Linkerd to provide mutual TLS between services, circuit breaking, advanced traffic management (canary deployments, traffic mirroring), and distributed tracing without application code changes.

**Centralized Logging.** Deploy Loki for centralized log aggregation, search, and alerting across all services, complementing the already-deployed Prometheus and Grafana stack.

**Event-Driven Communication.** Introduce a message broker (RabbitMQ or Apache Kafka) for asynchronous inter-service communication, reducing temporal coupling for non-critical operations such as notification dispatch and audit logging.

**TLS Termination.** Add TLS certificates (via cert-manager with Let's Encrypt) to the kGateway listener, encrypting all external traffic. Implement mutual TLS for internal service-to-service communication.

**Database Authentication.** Enable MongoDB authentication with the `MONGO_INITDB_ROOT_USERNAME` and `MONGO_INITDB_ROOT_PASSWORD` credentials that are already stored in Kubernetes Secrets but not yet enforced.

**Automated Testing.** Add integration and end-to-end test stages to the CI pipeline. Consider contract testing between the BFF and downstream services.

**Multi-Region Deployment.** Extend the architecture to support multi-region deployment for disaster recovery and global latency reduction.

## 19. Architecture Decisions (ADR)

### ADR-001: Microservices over Monolith

**Decision.** Decompose the application into eleven microservices (nine backend domain services, one BFF aggregator, and one frontend). **Rationale.** Independent deployability, technology heterogeneity potential, team autonomy, and fault isolation. **Trade-off.** Increased operational complexity, network latency between services, and the need for distributed transaction management.

### ADR-002: BFF Pattern over API Gateway

**Decision.** Use a Backend-for-Frontend service instead of a generic API gateway for request aggregation. **Rationale.** The BFF provides request-specific aggregation (role-based dashboard), a single authentication enforcement point, and simplifies the frontend's API surface to a single base URL. **Trade-off.** The BFF becomes a potential bottleneck and single point of failure for all API traffic.

### ADR-003: Gateway API over Ingress

**Decision.** Use the Kubernetes Gateway API (kGateway) instead of the traditional Ingress resource. **Rationale.** Gateway API is the successor to Ingress, offering richer routing capabilities, role-oriented resource model, and broader community adoption. **Trade-off.** Requires CRD installation and a compatible controller (kGateway).

### ADR-004: NFS over hostPath for Persistent Storage

**Decision.** Use NFS-backed dynamic provisioning instead of hostPath volumes. **Rationale.** NFS decouples storage from specific nodes, allowing pods to be rescheduled to any node without data loss. Dynamic provisioning eliminates manual PV management. **Trade-off.** Introduces dependency on the NFS server's availability and network performance.

### ADR-005: kubeadm over Managed Kubernetes

**Decision.** Use kubeadm on EC2 instances instead of a managed service (EKS, GKE). **Rationale.** Educational purpose -- hands-on learning of cluster bootstrapping, control plane components, and infrastructure management. **Trade-off.** Requires manual cluster maintenance, upgrades, and HA configuration.

### ADR-006: GitOps with ArgoCD over Direct Deployment

**Decision.** Use ArgoCD watching a manifest repository instead of direct `kubectl apply` from CI. **Rationale.** Git provides an auditable, versioned source of truth for cluster state. ArgoCD enables drift detection and self-healing. Separation of CI (build) and CD (deploy) concerns. **Trade-off.** Adds latency between code push and deployment (ArgoCD polling interval).

## 21. Release and Versioning Strategy

### 21.1 Semantic Versioning

All service images follow semantic versioning (vMAJOR.MINOR.PATCH). The CI pipeline automatically determines the version bump based on commit message prefixes. Commits with `feature:` trigger a minor version bump. Commits with `breaking:` trigger a major version bump. All other commits trigger a patch bump. Initial versions start at 1.0.0.

### 21.2 Image Tagging Strategy

Each build produces three image tags pushed to Docker Hub. The semantic version tag (e.g., `v1.2.3`) provides the stable, immutable reference. The SHA-suffixed tag (e.g., `v1.2.3-abc1234`) enables tracing an image to its exact source commit. The `latest` tag provides a convenience reference for development purposes.

### 21.3 Helm Chart Versioning

The umbrella Helm chart maintains its own version (`2.0.0` as of Chart.yaml), independent of individual service versions. The `appVersion` field tracks the platform release as a whole.

### 21.4 Independent Service Release

Services are released independently. Only the modified service's CI pipeline runs. The manifest repository tracks per-service image tags, allowing services to be at different version numbers simultaneously.

## 22. Incident Management and SLA

### 22.1 Monitoring and Alerting

The CI/CD pipeline includes email-based alerting at every stage. Pipeline failures trigger immediate email notifications to the configured alert address, including the service name, failure stage, branch, commit SHA, and a direct link to the failed workflow run.

### 22.2 Failure Recovery Procedures

**Service-Level Failure.** Kubernetes restarts failed pods automatically based on liveness probe failures. If a deployment update causes failures, rollback with `kubectl rollout undo` to restore the previous working version.

**Database Failure.** MongoDB ReplicaSet provides automatic failover. If the PRIMARY pod fails, a SECONDARY is automatically elected as the new PRIMARY. Applications reconnect via the replica set connection string.

**Infrastructure Failure.** If a worker node fails, Kubernetes reschedules pods to the remaining worker. MongoDB pods can tolerate one node failure due to pod anti-affinity across nodes.

### 22.3 SLA Targets (Recommended)

| Metric | Target |
|--------|--------|
| API Availability | 99.9% (monthly) |
| Incident Response Time | < 15 minutes (P1), < 1 hour (P2) |
| Mean Time to Recovery | < 30 minutes |
| Deployment Success Rate | > 95% |
| Change Failure Rate | < 15% |

## 23. Cost and Resource Optimization

### 23.1 Resource Budgeting

Based on the defined resource requests, the minimum cluster resource requirements are calculated as follows.

| Component | Instances | CPU Request (total) | Memory Request (total) |
|-----------|----------|--------------------|-----------------------|
| Backend Services (9 x 2 replicas) | 18 pods | 1800m | 2304Mi |

### 23.2 Optimization Recommendations

**Right-Size Instances.** Select EC2 instance types that match the aggregate resource requirements with approximately 20 percent headroom for burst capacity.

**Spot Instances for Workers.** Use EC2 Spot Instances for worker nodes to reduce compute costs by up to 90 percent, with pod disruption budgets configured to handle spot interruptions gracefully.

**Resource Limit Tuning.** Monitor actual resource consumption using Metrics Server or Prometheus and adjust requests and limits based on observed usage patterns.

**Image Size Reduction.** The multi-stage build pattern already reduces frontend image size. Backend services could further benefit from `.dockerignore` optimization and removing unnecessary files from the build context.

## 24. Compliance and Governance

### 24.1 Configuration as Code

All infrastructure and application configuration is stored in version-controlled Git repositories. This provides a complete audit trail of who changed what, when, and why through commit history.

### 24.2 Approval Workflows

The CI/CD pipeline includes a mandatory human approval gate before production deployment. This enforces separation of duties between code authoring and deployment authorization, aligned with change management best practices.

### 24.3 Security Scanning

Automated security scanning at two levels (dependency and container image) runs on every code change, providing continuous vulnerability assessment. Scan reports are retained as GitHub Actions artifacts for compliance review.

### 24.4 Recommendations for Further Compliance

**Secret Rotation.** Implement automated rotation for JWT secrets and database credentials using external secret management tools (such as AWS Secrets Manager or similar solutions).

**Audit Logging.** Enable Kubernetes audit logging and application-level audit trails for all administrative actions.

**Data Encryption at Rest.** Enable MongoDB encryption at rest and NFS volume encryption for data protection.

**Network Encryption.** Implement mutual TLS for all service-to-service communication using a service mesh.

## 25. Conclusion

MediMesh represents a comprehensive implementation of modern DevOps and microservices practices applied to the healthcare management domain. The platform demonstrates mastery of several critical engineering disciplines: microservices decomposition with clear bounded contexts, containerization with security-hardened images, Kubernetes orchestration with production-grade configurations, GitOps-based continuous deployment with ArgoCD, multi-stage CI pipelines with security scanning and approval workflows, and infrastructure management using Infrastructure as Code principles.

The architecture balances several competing concerns. Service independence is achieved through dedicated databases per service while maintaining data consistency through the BFF aggregation pattern. High availability is delivered through ReplicaSets, StatefulSets, and anti-affinity scheduling without the complexity of a multi-cluster setup. Security is enforced at multiple layers -- application (JWT/bcrypt), platform (RBAC/Secrets), network (NetworkPolicies), and pipeline (Snyk/Trivy scanning) -- while maintaining operational simplicity.

The platform is positioned for production use with the recommended enhancements outlined in Section 17, particularly the addition of TLS termination, a comprehensive observability stack, MongoDB authentication, and a staging environment. These improvements would elevate the platform from its current strong educational and demonstration posture to a fully production-hardened system suitable for enterprise healthcare operations.
